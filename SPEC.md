# ainput SPEC

## 1. 项目目标

从零开始设计并实现一个 Rust 主导的 Windows 本地语音输入工具 `ainput`。

它当前首版的目标不是翻译，也不是系统级输入法，而是：

- 让用户按住语音热键即可开始录音
- 松开语音热键后结束录音并触发本地离线 ASR
- 将识别结果直接插入当前 IDE / 浏览器 / 聊天输入框
- 当自动插入失败时，至少稳定写入剪贴板
- 让用户按截图热键进入截图冻结框选态
- 框选完成后将截图直接复制到剪贴板
- 可选在每次截图后额外自动保存 PNG 到桌面

## 2. 产品定位

- 产品名：`ainput`
- 类型：Windows 本地常驻工具
- 用户：项目作者本人
- 优先级：轻量、快速、稳定、自用效率优先

## 3. 技术路线

- 主语言：Rust
- ASR 模型：SenseVoiceSmall
- 推理接入：sherpa-onnx Rust API
- 兜底：sherpa-onnx C API
- 运行时：尽量只保留 Rust + sherpa-onnx 相关动态库

## 4. 当前设计口径

### 4.1 正式支持语种

- 中文
- 粤语
- 英语
- 日语
- 韩语

### 4.2 实际优化重点

- 中文口述输入
- 中英混合技术内容的基础识别可用性
- Windows 当前输入区域的直接粘贴体验

## 5. 本期必须项

- 用 Rust 重新设计项目骨架
- 建立可运行的 Cargo workspace
- 打通语音热键 -> 录音 -> ASR -> 文本输出主链路
- 实现自动粘贴与剪贴板输出
- 实现最小失败降级策略
- 启动后提供可见的托盘常驻入口
- 在按住录音期间提供屏幕下方、任务栏上方的底部悬浮条反馈
- 实现截图热键触发的冻结框选与图片剪贴板输出
- 在托盘右键菜单提供“截图后自动保存到桌面”开关
- 至少在一个 IDE 输入框和一个浏览器输入框中验证可用
- 提供可安装、可卸载的 Windows 安装包
- 安装后有明确单一入口，不再依赖用户手动解压后区分 `exe` 与 `bat`

## 6. 本期不做

- 系统级 IME 注册
- 在线大模型作为主链路
- 原项目 Python UI 兼容
- 跨平台适配
- 插件系统
- 多用户同步
- 自动生成提示词
- 术语增强第一版
- 模板 / 场景 / 账本系统
- 截图编辑器
- OCR
- 滚动截图
- 保存目录自定义

## 7. 关键约束

- 首版必须围绕自用效率，不追求大而全
- 最终运行时尽量不依赖 Python
- UI 必须极简
- 当前阶段不让后续增强能力阻塞主链路交付
- 所有任务推进默认回写本目录文档
- 截图功能必须使用 Rust 接 Windows 底层 Windows API 实现屏幕采集与选区交互

## 8. 验收标准

### 8.1 功能验收

- 按住语音热键可启动录音
- 松开语音热键可停止录音并触发本地离线识别
- 识别结果可直接进入当前输入框
- 自动插入失败时可稳定进入剪贴板
- 启动后用户能在托盘确认程序仍在运行
- 按住录音时用户能看到屏幕下方、任务栏上方的底部悬浮条，松开后悬浮条消失
- 按截图热键后用户能进入冻结的全屏框选截图状态
- 松开鼠标后截图能直接进入剪贴板
- 托盘勾选“截图后自动保存到桌面”后，每次截图都会额外保存 PNG 到桌面
- 取消勾选后，后续截图不再自动保存
- 至少在一个 IDE 输入框和一个浏览器输入框里完成实测
- 可生成一个可双击安装的 Windows 安装包
- 安装后程序文件能落到固定目录
- 安装后用户可通过开始菜单或安装目录中的单一主程序入口启动
- 卸载后安装目录、开始菜单入口、卸载注册信息可清理

### 8.2 架构验收

- Rust 为主程序语言
- sherpa-onnx 为 ASR 运行时核心
- Python 不进入最终常驻主链路
- 项目目录职责清晰
- 工作流、日志、任务清单齐全

## 9. 当前重构专项目标

本轮不再把 `ainput` 视为“功能已基本完成，只修几个点”，而是视为一次围绕长期自用稳定性的专项重构。

### 9.1 重构目标

- 把语音输入链路从“能跑”提升到“稳定、可解释、可配置、可维护”
- 把截图链路从“新增功能”提升到“和语音链路并列的一等能力”
- 把当前散落在托盘、配置、词表、日志里的行为重新收口成清晰产品结构
- 把依赖模拟输入、轮询 hook、隐式学习、模糊状态的实现，尽量替换为 Windows 原生接口或可验证的显式机制

### 9.2 本轮必须覆盖的问题

- 文中插入句号省略规则在 Codex CLI / 终端类输入区失效
- 自动学习机制不可观察、不可确认、不可迁移，且不符合 AI 编程场景
- 增加“语音触发 emoji”能力：当光标位于末尾时，句尾口述 `笑死` 需要输出为 `[破涕为笑]`
- 新增语音历史滚动记录能力，默认保留最近 500 条，一行一条
- 托盘菜单重构为按“语音 / 截图 / 通用”分组的产品级信息架构
- 热键改为可配置，并优先接入 Windows 原生热键注册接口
- 语音与截图能力都要重新核验是否已采用最稳定的 Windows 原生链路；能原生则原生
- 修复当前已知实现风险：截图剪贴板句柄所有权、伪配置项、根目录发现过宽、空闲高频 tick、发布版本信息分散
- 后台日志、历史、学习等维护动作必须与前台语音识别和截图主链路解耦，不能阻塞主程序
- 程序必须保证单实例运行；若启动第二个实例，应先结束旧实例，再由新实例接管

### 9.3 新的产品口径

- `ainput` 是一个“语音输入 + 截图采集”的双链路常驻工具
- 两条链路平级存在，共享托盘、配置、日志和运行时，但各自应有独立状态、独立设置、独立验收
- 轻量设置优先采用本地文本配置文件；只有当托盘无法承载必要认知时，才补一个极轻量设置面板
- 面向 AI 编程场景做默认优化，尤其是高频英文技术词、品牌词、模型词、工具词

### 9.4 本轮完整验收标准

- 在普通输入框与终端类输入框中，句号策略都有可解释行为；无法判断上下文时不应强行补句号
- 术语与学习系统变为“可见、可审计、可导入、可迁移”的资产，而不是隐式文本猜测
- 在普通可编辑输入框的行尾口述“笑死”时，最终输出应为句尾 `[破涕为笑]`
- 当光标不在末尾、句中仍有内容、或处于终端类 / 未知上下文时，不应误触发 emoji 替换
- 首次安装后即自带一套面向 AI 编程的高频术语资产，而不是几乎空白
- 每次语音识别都能追加写入历史文件，并自动滚动裁剪到最近 500 条
- 托盘菜单信息结构清晰，用户能一眼区分语音、截图、通用入口
- 热键可配置，且注册、冲突、失败原因都可见
- 语音与截图链路中需要接管系统能力的部分，优先切换到 Windows 原生接口；不能切换的部分必须明确原因
- 修复后至少补齐关键自检命令、日志与最小自动化验证，避免再次回到“只能手猜是否生效”
- 后台维护动作不会阻塞语音识别与截图主链路
- 启动第二个实例时，最终系统内只保留一个 `ainput-desktop.exe`
