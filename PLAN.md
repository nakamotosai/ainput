# ainput PLAN

## 阶段 0：项目启动

目标：
把项目起步所需的目录、规则、决策、计划和日志体系搭好。

输出：

- Cargo workspace 骨架
- AGENTS / SPEC / PLAN / TASKLIST / ARCHITECTURE / DECISIONS / WORKFLOW / OPLOG

验证：

- 新会话进入目录后可直接开始开发

## 阶段 1：Rust 基础骨架

目标：
建立 Rust workspace 与首版所需模块边界，不写完整业务，只先搭载体。

动作：

- 建立 workspace
- 建立主应用 crate
- 建立 shell / audio / asr / rewrite / output / data crate
- 建立配置与日志基础设施

验证：

- `cargo check` 可跑通最小骨架

## 阶段 2：ASR 主链路

目标：
打通 Rust -> sherpa-onnx -> SenseVoiceSmall 的最小识别链路。

动作：

- 选定 Rust API 或 C API 实际落点
- 设计模型目录约定
- 先建立 wav 文件到文本的最小闭环
- 再建立麦克风录音到文本的最小闭环

验证：

- 录音后可得到文本结果

## 阶段 3：输入控制与输出注入

目标：
实现一个能真正服务日常使用的最小常驻语音输入工具。

动作：

- `Alt+Z` 全局热键
- 录音状态机
- 剪贴板
- 自动粘贴
- 插入失败降级

验证：

- 至少在一个 IDE 和一个浏览器输入框中可用

## 阶段 4：极简 UI 与托盘

目标：
只保留必要交互，不做大窗体产品。

动作：

- 托盘
- 录音状态反馈
- 最近结果预览
- 最小设置入口

验证：

- UI 不干扰主流程

## 阶段 4.5：截图链路

目标：
把 `ainput` 从单一语音输入工具扩成“语音 + 截图”的双链路常驻工具。

动作：

- 新增 `Alt+X` 全局截图热键
- 实现整屏冻结与框选窗口
- 实现图片剪贴板输出
- 实现托盘里的截图自动保存开关
- 实现桌面 PNG 自动保存

验证：

- 截图可复制到剪贴板
- 勾选时可保存到桌面
- 关闭勾选后不再自动保存

## 阶段 5：后续增强能力

目标：
在主链路稳定后，再逐步补强识别后的文本质量与复用能力。

动作：

- 术语增强
- 提示词转换
- 高频场景模板
- 账本复用

验证：

- 主链路稳定后再进入单项增强验收

## 阶段 6：打包与回归

目标：
形成可长期自用的可运行版本。

动作：

- 打包
- 安装包
- 模型部署说明
- 回归测试
- 操作日志回写

验证：

- 本机可持续日用

## 阶段 9：稳定性与产品化重构

目标：
把现有“语音 + 截图”原型收敛为长期可用的自用工具，优先解决可解释性、可配置性、系统接口稳定性和状态漂移问题。

动作：

### 9.1 输入上下文与标点策略重构

- 重新梳理“当前光标是否在行尾 / 文中 / 终端输入区”的判断模型
- 将当前基于 UI Automation 的能力分层：
  - 第一层：原生文本接口可判断上下文时，继续用原生接口
  - 第二层：终端 / 控制台 / 类 TTY 输入区无法判断时，切到保守模式，不自动补句号
  - 第三层：必要时补充 Windows 原生可用的前台窗口类别识别，而不是只看 UIA
- 明确“未知上下文”的默认行为与日志签名

验证：

- 普通编辑器输入框
- 浏览器输入框
- Windows Terminal / Codex CLI / 终端类输入区
- 日志中能看出本次命中了哪一类上下文策略

### 9.2 术语与学习系统重构

- 废弃当前“托盘点一下，从最近修正和剪贴板猜一个词”的主机制
- 改为三层资产：
  - 内置 AI 编程词库
  - 用户显式词库
  - 学习候选池 / 待确认池 / 已激活映射
- 为学习系统增加显式状态：
  - 新候选
  - 已累计
  - 已激活
  - 已忽略 / 已禁用
- 提供可见反馈入口：
  - 托盘里的“最近学习结果”
  - 本地学习日志
  - 可直接打开的结构化词表文件
- 预置一套面向 AI 编程场景的高频英文词

验证：

- 首次安装后不需要手工重建基础词库
- 学习动作成功、失败、未命中都有明确反馈
- 用户能看见哪些词已生效，哪些只是候选

### 9.2.1 语音触发 emoji

- 为识别后文本增加一层轻量“语音触发动作”规则
- 首版只支持：句尾口述 `笑死`，在光标位于末尾时替换为 `[破涕为笑]`
- 规则依赖现有输出上下文判断：
  - `EditableAtEnd` 允许触发
  - `EditableWithContentOnRight` 禁止触发
  - `Unknown` 默认禁止触发
- 先采用固定内置规则，不在本轮引入通用 DSL 或复杂可配置系统
- 触发顺序固定为：
  - 先做基础文本归一化
  - 再判断输出上下文
  - 再执行 emoji 替换
  - 最后执行现有句号补全 / 去除策略

验证：

- 行尾输入框中口述“笑死”后输出为 `[破涕为笑]`
- 句中“笑死”不被替换
- 未知上下文不触发该规则
- 现有句号策略不被破坏

### 9.3 语音历史滚动日志

- 新增一份专用历史文件
- 记录格式固定为“一行一条”
- 记录内容至少包含：
  - 时间戳
  - 最终输出文本
  - 可选的输出类型或结果标签
- 每次写入后滚动裁剪，只保留最近 500 条
- 明确和 `last_result.txt` 的职责边界

验证：

- 连续写入后文件内容稳定
- 超过 500 条时正确裁剪
- 文件可直接用记事本打开查看

### 9.4 托盘信息架构与设置模型重构

- 按“语音 / 截图 / 通用”重做托盘菜单分组
- 将状态显示与操作入口分离
- 识别哪些设置适合托盘即时切换，哪些应进入文本配置或设置面板
- 优先采用极客型文本配置文件
- 仅当文本配置无法承载必要可用性时，再补轻量设置面板

验证：

- 托盘一眼能区分两条主能力
- 常用动作不超过两层认知跳转
- 低频复杂项不再挤进托盘主菜单

### 9.5 热键系统重构

- 放弃当前依赖低层键盘状态机硬拦截作为主方案
- 语音热键与截图热键优先切到 Windows 原生热键注册接口
- 设计冲突检测、注册失败提示、配置落盘、热更新与回退策略
- 对鼠标中键长按单独处理，不与主热键模型混在一起

验证：

- 热键可改
- 热键冲突或注册失败时有明确反馈
- 主热键不再依赖复杂 Win 键透传状态机维持

### 9.6 系统接口与实现归一化

- 重新审计语音输出、截图采集、截图剪贴板、热键注册分别走的系统接口
- 能改用 Windows 原生接口的地方优先改
- 不能改的地方记录明确技术理由与回退路径
- 修复截图剪贴板句柄所有权问题

验证：

- 每个关键能力都有清晰的接口选型说明
- 截图复制链路在失败时不会留下悬空句柄或半成功状态

### 9.7 运行时与发布治理

- 清理伪配置项与已失效配置语义
- 收紧根目录发现逻辑，避免静默落到错误 cwd
- 降低空闲 overlay tick 资源消耗
- 收口发布版本号来源，避免脚本与文档分散硬编码
- 补齐 `fmt` / `clippy` / 关键自检命令

验证：

- `cargo fmt --check`
- `cargo clippy --all-targets --all-features -- -D warnings`
- 关键自检命令与文档一致

### 9.8 后台维护与前台主链路解耦

- 将最近结果与语音历史的持久化统一放到独立维护线程
- 前台语音输出和截图完成后立即返回，不等待后台写文件
- 移除周期性资源心跳，避免常驻维护动作抢占前台体验
- 保留必要日志，但不再让维护日志成为默认高频动作

验证：

- 识别完成时可直接进入输出路径
- 历史文件可随后异步写入
- 截图复制路径不依赖后台维护线程

### 9.9 单实例接管

- 在桌面常驻模式启动前检测已有 `ainput-desktop.exe`
- 若已有旧实例，则由新实例主动结束旧实例
- 仅在确认旧实例退出后继续启动新的托盘常驻
- 调试命令和自检命令不参与单实例接管，避免互相干扰

验证：

- 先启动一份，再启动第二份
- 最终系统内只保留一个常驻实例

### 9.10 截图模式视觉反馈

- 在现有全屏截图窗口内直接绘制半透明黑色遮罩，不新增额外窗口
- 遮罩强度控制在约 28%，保证“明显进入截图态”但不过黑
- 选区内重绘原始截图，形成“内亮外暗”的视觉结构
- 选区边框使用 1px 白色描边，鼠标继续保持十字
- 取消或完成截图时直接销毁窗口，不做淡入淡出动画

验证：

- 触发截图后，整屏立即变暗
- 拖选时选区保持原亮度，边框清晰可见
- 完成或取消截图后，遮罩立即消失
