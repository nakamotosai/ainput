# HUD 微流式吐字平滑 SPEC

> 状态：已被 `../streaming-realtime-rewrite-v3/` 取代。本文只保留历史背景；继续开发和验收以 V3 `SPEC.md` / `PLAN.md` / `TASKLIST.md` 为准。

## 目标

- 解决当前流式模式里“HUD 上字是一块一块冒出来，不够丝滑”的体感问题。
- 保持现有 `preview.9` 的稳定基线不变：
  - 识别不中断
  - 松手上屏仍然快
  - 长句不会重新退化成“只出前几个字”

## 非目标

- 不把主链重新切回真 streaming zipformer worker。
- 不为了追求“一字一字吐出”而重新引入高风险识别架构。
- 不改 HUD 视觉皮肤，不做样式重设计。
- 不牺牲当前已经改善好的最终上屏速度。

## 当前事实

- 当前流式模式虽然已经稳定，但 HUD 文本更新节奏仍由 ASR 预览周期驱动。
- 预览周期当前是自适应节流：
  - `0-3s`: `250ms`
  - `3-8s`: `400ms`
  - `8s+`: `650ms`
- 每轮预览都会生成一段新的整理文本，主线程收到 `StreamingPartial` 后，HUD 直接整段替换。
- 因此用户看到的不是“连续吐字”，而是“隔一段时间，几个字一起跳出来”。

## 根因判断

- 这不是 HUD 渲染性能问题，主要是“识别结果生成节奏”和“HUD 呈现节奏”完全绑定。
- 即使继续只压识别周期，HUD 仍然会表现成“块状刷新”，只是块变小、CPU 变高。
- 真正缺的是一层“HUD 呈现补间层”：
  - 识别仍按当前稳定节奏产出
  - HUD 不再直接整段替换，而是把新增内容逐字展开

## 推荐方案

### 总体思路

- 保持当前 ASR worker 的稳定主链不动。
- 在 HUD 层新增“目标文本”和“当前显示文本”之间的逐字动画器。
- 每当收到新的 `StreamingPartial.prepared_text`：
  - 先和当前 HUD 目标文本做差异分析
  - 找到稳定前缀
  - 对新增后缀做严格逐字补间显示
- 如果新结果对旧结果有修正：
  - 稳定前缀之前的内容不动
  - 从变化点开始立即替换到新目标，再继续逐字展开剩余部分
- 用户机器性能允许，因此本任务直接把预览周期整体压短一档，不做渐进试探。

### 为什么选它

- 风险最低：不碰主识别链架构。
- 体感最直接：用户看到的就是 HUD 更新方式。
- 成本可控：主要改 `main.rs` 的事件消费层、`overlay.rs` 的 HUD 驱动层，以及 `worker.rs` 的节流参数。
- 这轮可以一次完成，不需要再分“先 UI 再提频”的长链路。

## 方案分层

### 方案 A：只降低 ASR 预览周期

- 做法：
  - 直接把 `0-3s` 压到 `120-160ms`
  - 其余阶段也同步下调
- 问题：
  - CPU 和卡顿风险会直接上升
  - HUD 仍然是“块状刷新”，只是块变小
- 结论：
  - 不单独采用，但会作为主方案的配套增强一起上线

### 方案 B：重新换真 streaming partial 模型

- 做法：
  - 改用真正擅长 token 级流式输出的小模型
- 问题：
  - 这是重新走回高风险路线
  - 之前已经证明这条路极易复发“只出前几个字”
  - 会把当前稳定基线重新打散
- 结论：
  - 当前版本不做

### 方案 C：HUD 逐字微流式补间层

- 做法：
  - 识别输出仍块状
  - HUD 呈现改成逐字连续展开
- 优点：
  - 风险最小
  - 改动集中
  - 用户体感提升最明显
- 结论：
  - 作为本任务主方案

### 方案 D：HUD 逐字微流式补间层 + 直接提频

- 做法：
  - HUD 改成严格逐字吐字
  - 预览周期直接改成：
    - `0-3s`: `120ms`
    - `3-8s`: `220ms`
    - `8s+`: `360ms`
- 结论：
  - 作为本任务最终落地方案

## 详细设计

### 1. HUD 文本状态从“单文本”升级为“双缓冲”

- 当前需要新增两层状态：
  - `hud_target_text`
    - 最新一次来自 worker 的整理文本
  - `hud_display_text`
    - 当前真正显示在 HUD 上的文本

- HUD 不再每次收到 `StreamingPartial` 就直接 `set_text(prepared_text)`。
- 改成：
  1. 更新 `hud_target_text`
  2. 由 HUD 动画器在后续 `tick()` 中逐字逼近 `hud_target_text`

### 2. 差异算法

- 使用“最长公共前缀”作为保守同步点。
- 场景处理：

- 纯新增：
  - 旧：`帮我看一下`
  - 新：`帮我看一下这个问题`
  - 处理：稳定前缀是整句旧文本，新增部分 `这个问题` 逐字进入 HUD

- 后缀修正：
  - 旧：`帮我看一下这里有没`
  - 新：`帮我看一下这里有没有问题`
  - 处理：前缀 `帮我看一下这里有` 稳定，后面立即替换为新后缀，再继续逐字展开

- 整段收缩或明显改写：
  - 处理：从变化点开始直接切到新目标，不做慢速逐字回删，避免 HUD 来回抖动

### 3. 吐字节奏

- 默认节奏：
  - HUD 每 `12-18ms` 追加 1 个字符
  - 不按小段推进，严格逐字吐出

- 规则：
  - 中文、英文、数字统一按字符推进
  - 不再做“小块 token”优化
  - 标点和空格也按字符推进

### 4. 追帧策略

- 如果新 `hud_target_text` 来得太快，而动画器还没追上：
  - 不把旧队列慢慢播完
  - 直接和最新目标重新做 diff
  - 从最新稳定前缀开始继续逐字输出

- 目的：
  - 避免 HUD 落后真实识别太久
  - 保证“看起来连续”但不拖尾

### 5. 松手收尾策略

- `StreamingFinal` 到来时：
  - 立即把 `hud_display_text` 对齐到最终文本
  - 停止逐字动画
  - 按当前已有逻辑显示 `已识别`
  - 保持最终上屏速度优先

### 6. 日志

- 增加 HUD 呈现层日志，和识别层分开：
  - `hud_microstream_started`
  - `hud_microstream_retargeted`
  - `hud_microstream_flushed_to_final`
  - `hud_display_chars`
  - `hud_target_chars`
  - `hud_diff_prefix_chars`

- 这样以后如果用户反馈“HUD 还是跳块”，可以先分辨：
  - 是识别本身出字慢
  - 还是 HUD 动画器没有正确工作

## 轮次规划

### Round 1：HUD 逐字渲染层

- 新增 `hud_target_text / hud_display_text`
- 新增差异计算和逐字推进状态
- 新增 `overlay.tick()` 驱动的逐字推进
- 保持识别主链不动

### Round 2：异常修正与收尾

- 处理后缀修正、文本回退、最终文本强制对齐
- 避免 HUD 在修正时闪烁或倒退得很奇怪
- 增加日志和最小纯逻辑测试

### Round 3：直接提频并联调

- 直接把识别预览周期压短
- 联合观察：
  - HUD 逐字吐字是否跟得上
  - CPU 是否可接受
  - 长句后半段是否仍稳定

## 验收标准

- 用户连续说一句长话时，HUD 的文字体感应明显比现在更连续，不再是大块跳字。
- HUD 的新增文字必须严格逐字出现，而不是几字一组。
- 修正发生时，HUD 不应出现整段闪烁、整句回退、疯狂抖动。
- 松手最终上屏速度不能明显变慢。
- 长句稳定性不能退回老问题。

## 主要风险

- 风险 1：HUD 虽然更丝滑，但和真实识别结果脱节太多
  - 控制：动画器始终追最新目标，不允许队列无限落后

- 风险 2：修正时出现“先显示错字，再慢慢改回来”的违和感
  - 控制：对删改场景优先立即替换变化段，只对新增部分做逐字展开

- 风险 3：提频后 CPU 占用上升
  - 控制：先限定在当前三档固定周期，不继续无限压缩

## 结论

- 这件事最合理的做法，不是换模型，而是：
  - 保持当前识别主链
  - 在 HUD 层加严格逐字吐字的微流式渲染
  - 同时把预览周期直接压短一档
